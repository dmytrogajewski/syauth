# THREAT-2026-05-15: syauth v0.1 close-out threat model

> Skill: `.agents/skills/threat/SKILL.md` Phase 1 through Phase 7.
> Roadmap item: `specs/syauth/ROADMAP.md` **S-020**.
> Journey: [specs/journeys/JOURNEY-S-020-threat-closeout.md](../journeys/JOURNEY-S-020-threat-closeout.md).
> Spec cross-reference: `specs/syauth/SPEC.md` §6 (executive summary,
> T-001..T-010) and §4 (dataflow, wire format, replay cache).

## 0. Summary

syauth is a **phone-presence-plus-consent** Linux PAM authenticator
that uses BLE GATT as the primary transport and a mandatory
per-unlock biometric gesture on the bonded Android phone as the
defeater for the entire NCC-class BLE relay attack family. This
document closes the v0.1 threat model against the actually-shipping
code at commit `c61638d` (S-001 through S-019 complete). Every one of
the ten canonical proximity-unlock abuse paths is either
**mitigated** (cited by file:line plus a passing test) or
**accepted-residual** (cited by rationale paragraph). No path remains
`open`. Two genuine residuals (T-007 root-on-host, T-001 partial:
hardware-aided relay rigs that synthesise a fresh biometric) are
called out explicitly; both are explicitly out-of-scope for v0.1 per
SPEC §3.4.

## 1. Scope

### 1.1 In scope

- `pam_syauth.so` cdylib (crate `syauth-pam`, including the
  `pam_sm_authenticate` / `pam_sm_setcred` / `pam_sm_acct_mgmt`
  entry points and the `catch_unwind` panic boundary).
- `syauth-cli` desktop binary (`pair`, `list`, `revoke`, `status`,
  `install-pam`, `uninstall-pam` subcommands).
- BLE GATT central transport (crate `syauth-transport`, `BlueZBtPeer`
  via `bluer`; mock peer for unit and integration tests).
- `syauth-core` shared protocol (frame format, replay cache, MAC,
  Ed25519 signing, bond store, secret-storage trait + Linux
  backends).
- Android companion (`syauth-android` Gradle module, `syauth-mobile`
  UniFFI surface, `SyauthCompanionService`, BiometricPrompt-gated
  Keystore signer).
- Configuration file at `/etc/syauth.conf` and bond store at
  `/var/lib/syauth/bonds.toml`.
- The wire-format frame `[ver=1][nonce:16][payload][tag:16]`.

### 1.2 Out of scope (but compromise affects us)

- The Linux kernel and BlueZ stack itself.
- The Android OS lockscreen, including the BiometricPrompt
  implementation.
- Physical security of the desktop chassis (a thief who can power-off
  the host can hold the unencrypted bond key file at rest if the host
  uses an unencrypted root partition — this is out-of-scope by the
  shape of disk-encryption policy).
- Other PAM modules in the stack — syauth deliberately preserves
  fallback to `pam_unix` for the "phone is dead" recovery path
  (SPEC D7); a misconfigured stack that puts a too-weak module before
  syauth is a deployment, not a protocol, issue.
- A coerced biometric prompt where the attacker holds the user and
  forces the tap (this is a rubber-hose attack and is named, not
  defeated, by syauth).

## 2. Assets, Actors, Trust Boundaries

### 2.1 Assets

| Asset | Why it matters | Storage |
|-------|----------------|---------|
| Bond key (per peer, 32 bytes) | Forging it bypasses every unlock from that peer | Kernel keyring on Linux; Android Keystore on the phone |
| Phone's Ed25519 signing key | Sole proof that the phone is who it claims | Android Keystore with `setUserAuthenticationRequired(true)` |
| Host's Ed25519 signing key (challenge author) | Stops a hostile actor from forging a desktop's challenge | Kernel keyring; uploaded from `/etc/syauth/host.key` (0600 root) at boot |
| Bond record (peer pubkey, name, status) | Loss = lost recovery + revocation surface | `/var/lib/syauth/bonds.toml` (0600 root) |
| Pairing-time numeric code | Predicting it lets an attacker bond a rogue phone | Ephemeral, displayed once on both sides; never persisted |
| Replay nonce cache | Drops replayed responses inside a session | In-memory only, dropped with the per-call tokio runtime |
| Session UUID | Advertises the host without leaking a stable id | Rotated every minute by HKDF over the bond key |

### 2.2 Actors

| Actor | Capability bound |
|-------|------------------|
| **Remote net attacker** | Can reach the phone over IP; cannot reach desktop BT |
| **Local radio attacker** | BLE radio within ~10 m of either device; can sniff and inject |
| **Relay attacker** | Two radios with low-latency link; one near phone, one near desktop |
| **Co-located local user** | Shell access on desktop as a different uid |
| **Root attacker** | Already root on the desktop |
| **Phone thief** | Has the phone, may or may not know the user's PIN |
| **Desktop thief** | Has the desktop, not the phone |

### 2.3 Trust boundaries

1. Air gap between phone and desktop (BLE GATT).
2. Process boundary between `pam_syauth.so` (loaded into login/sudo)
   and `syauth` CLI (run as user).
3. Storage boundary between kernel keyring / Android Keystore and
   on-disk config.
4. ABI boundary between Rust and libpam (the only `unsafe` Rust call
   sites in the workspace; see `syauth-pam/src/entry.rs`).

## 4. Domain Abuse Paths (canonical ten)

The `/threat` SKILL §4 names these 4.1..4.10; SPEC §6 names the same
threats T-001..T-010. The mapping is identity (path 4.x ↔ T-00x).

### 4.1 / T-001 — Relay attack

**Path.** Attacker A holds a BLE radio near the phone; attacker B
holds a BLE radio near the desktop; A and B share a low-latency
back-channel. The desktop emits a freshly-signed challenge; B relays
it through to A, who replays it to the phone over the air; the phone
signs whatever it receives and ships the response back. End-to-end the
desktop sees a valid signed response and admits.

**Mitigation in v0.1.** Per SPEC §3.4 anti-goal and D2: *every*
response requires a fresh BiometricPrompt approval on the phone.
Without the user actively tapping Approve under their own
fingerprint / face / device PIN, the Android Keystore refuses to
release the signing key (SPEC D6 plus the `setUserAuthenticationRequired(true)`
contract in S-017). The relay therefore turns into "an attacker
forces the user to tap right now," which is not a passive-presence
attack any longer.

**Code citation.**
- `syauth-android/app/src/main/java/dev/syauth/companion/ApproveActivity.kt`
  hosts the BiometricPrompt and the
  `setUserAuthenticationRequired(true)` Keystore signer (S-017).
- `crates/syauth-pam/src/auth.rs:303-309` (response read-back inside
  a deadline-bounded tokio runtime).

**Test.**
- `crates/syauth-pam/tests/pam_e2e.rs::tc01_golden_scenario_returns_pam_success`
  (the round-trip works when the user approves).
- `crates/syauth-pam/tests/pam_e2e.rs::tc02_offline_scenario_returns_authinfo_unavail_under_budget`
  (no approval → no admit, plus latency budget).

**Status.** **Mitigated** for the passive-relay attack class. **Residual:**
a coerced-biometric / rubber-hose attacker, or a hardware setup that
synthesises a fresh biometric to the phone's sensor, still wins. Both
are documented in §7 as accepted-residual under T-001-RES.

### 4.2 / T-002 — Replay

**Path.** Attacker captures a successful response frame and replays
it later.

**Mitigation.** Per-call sliding-window replay cache. The PAM hot
path constructs a fresh `ReplayCache` with `DEFAULT_REPLAY_CAP = 64`
entries and `DEFAULT_REPLAY_TTL = 10 s` (SPEC §4.2), observes every
inbound nonce against it, and translates `Acceptance::Replayed` to
`AuthErr { reason: "replay" }`. Replays from a *previous* PAM
invocation are bounded by the fact that the cache lives only inside
the per-call tokio runtime — but every nonce is paired to a
challenge whose nonce the desktop just generated, so cross-call
replays would have to replay the desktop's *challenge* nonce too,
which the verifier rejects on signature mismatch.

**Code citation.**
- `crates/syauth-core/src/replay.rs:78-118` (`ReplayCache::new`
  / `observe`).
- `crates/syauth-pam/src/auth.rs:432-447` (cache instantiation,
  optional test seed, response-nonce observation, `AuthErr` mapping).

**Test.**
- `crates/syauth-core/src/replay.rs::tests::exact_replay_rejected`.
- `crates/syauth-core/src/replay.rs::tests::replay_does_not_refresh_inserted_at`.
- `crates/syauth-pam/tests/pam_e2e.rs::tc04_replay_returns_pam_auth_err`.

**Status.** **Mitigated.**

### 4.3 / T-003 — MitM during pairing

**Path.** Attacker interposes between phone and desktop during the
initial bond. If `Just Works` is used, the BT pairing exchange is
silently MitM-able.

**Mitigation.** Two layers:
1. The pair flow refuses to proceed on an adapter that does not
   advertise LE Secure Connections (LESC). The check runs before
   anything else and is unconditional (`--yes` does not skip it).
2. An app-level OOB confirmation derived from the negotiated bond
   key (`HKDF(bond_key, "syauth-oob-v1")[..4]` rendered as 4 emoji
   words) must be confirmed by the operator on both ends. An
   attacker who broke LESC would still need to spoof this
   independently-derived code.

**Code citation.**
- `crates/syauth-cli/src/pair.rs:444-459` (the LESC gate inside
  `run_pair_with_io`).
- `crates/syauth-cli/src/pair.rs:489-511` (the OOB confirmation
  step, `oob_code_for_bond` derivation, Accept / Reject branches).
- `crates/syauth-cli/src/oob.rs` (the `HKDF(bond, "syauth-oob-v1")`
  derivation).

**Test.**
- `crates/syauth-cli/tests/pair_flow.rs::pair_rejects_when_adapter_lacks_lesc`.
- `crates/syauth-cli/tests/pair_flow.rs::pair_rejects_when_adapter_lacks_lesc_even_with_yes`.
- `crates/syauth-cli/tests/pair_flow.rs::pair_operator_reject_writes_no_bond`.

**Status.** **Mitigated.**

### 4.4 / T-004 — Rogue device bonding

**Path.** Attacker triggers a pairing flow with their own phone
while the user is at the desk; the user, expecting their own pair
flow, confirms out of habit.

**Mitigation.** Pairing is exclusively *operator-initiated* via
`syauth pair` on the desktop. The pair flow is the only path that
reads from stdin, displays the BT numeric code and the 4-word
emoji OOB code, and writes a bond record on Accept. The PAM unlock
path does not accept inbound bond requests, and the bond store has
no "auto-add" path. An attacker who tricks the user into running
`syauth pair` still has to win the LESC numeric comparison (a fresh
6-digit code per pair flow) AND the app-level OOB confirmation.

**Code citation.**
- `crates/syauth-cli/src/pair.rs:439-523` (the only code path that
  writes a bond record on the desktop).
- `crates/syauth-pam/src/auth.rs:208-232` (the PAM auth path reads
  the bond store and refuses to mutate it; "no bonded peer" is a
  typed `AuthInfoUnavail`).

**Test.**
- `crates/syauth-cli/tests/pair_flow.rs::pair_operator_reject_writes_no_bond`
  (operator says N, no bond is written).
- `crates/syauth-cli/tests/pair_flow.rs::pair_timeout_leaves_pre_existing_bonds_file_byte_equal`
  (a timed-out pair flow does not perturb the existing bonds file).

**Status.** **Mitigated.**

### 4.5 / T-005 — PAM stack misconfiguration

**Path.** Operator edits `/etc/pam.d/sudo` and writes `auth sufficient
pam_syauth.so` (instead of `required`). If syauth returns
`PAM_AUTHINFO_UNAVAIL` (peer offline) the stack falls through to the
next module, but `sufficient` means an *upstream* admit short-circuits
the rest of the stack. With the wrong control flag, an attacker who
can produce a partial syauth success (e.g. via a bug elsewhere in the
module) bypasses the password.

**Mitigation.** The `syauth install-pam` helper hard-codes
`CONTROL_FLAG = "required"` and inserts the directive atomically
with a `.bak` snapshot. Operators are documented (in
`docs/security.md` and SPEC D7) on the `required` vs `sufficient`
semantics; the password fallback is preserved by leaving `pam_unix`
*after* the syauth line.

**Code citation.**
- `crates/syauth-cli/src/install_pam.rs:38` (`pub const CONTROL_FLAG: &str = "required"`).
- `crates/syauth-cli/src/install_pam.rs:34-45` (default args, field
  separator).
- `crates/syauth-cli/tests/install_pam.rs` (atomic-edit and idempotency
  asserts).

**Test.**
- `crates/syauth-cli/src/install_pam.rs::tests::*` covers the
  control-flag, format, and idempotency invariants in unit form;
  the integration tests in `crates/syauth-cli/tests/install_pam.rs`
  pin the atomic-write contract.

**Status.** **Mitigated by docs + tooling.** The residual is "operator
hand-edits the file later"; that is documented in
`docs/security.md`'s operational-hygiene checklist.

### 4.6 / T-006 — Phone-thief escalation

**Path.** Attacker steals the phone while the user is logged in to
their desktop. The attacker then approaches the desktop and unlocks
it via the stolen phone.

**Mitigation.** The Keystore-backed signing key is gated by
`setUserAuthenticationRequired(true)`; the key only releases under a
fresh BiometricPrompt success. An attacker holding the phone but
unable to defeat the lockscreen (fingerprint, face, or device PIN)
cannot get a signature. If the phone is unlocked at the moment of
theft, the attacker has the duration of one
"screen-on-and-Approve-tap" window — comparable to the existing
threat surface of any unlocked phone, and identical to the threat
surface of any phone-as-key system (Apple Auto Unlock, Bitwarden
push, Duo). syauth does not promise to be safer than the phone's own
lockscreen.

**Code citation.**
- `syauth-android/app/src/main/java/dev/syauth/companion/keystore/KeystoreSigner.kt`
  (S-017): the `setUserAuthenticationRequired(true)` flag is the
  Android-Keystore-level gate.
- `syauth-android/app/src/main/java/dev/syauth/companion/ApproveActivity.kt`
  (S-017): the BiometricPrompt is the user-facing gate on top of
  Keystore.

**Test.**
- `syauth-android/app/src/androidTest/java/dev/syauth/companion/KeystoreSignerInstrumentationTest.kt`
  (Robolectric-shaped, run by `make android-test`).
- The desktop-side e2e in `tests/e2e_real.rs` (gated on
  `SYAUTH_E2E_REAL=1`) drives this Android path on a real emulator.

**Status.** **Mitigated** by Android Keystore + BiometricPrompt.
**Residual:** a phone unlocked in the attacker's hand at the moment
of theft (T-006-RES); same residual the entire Android-ecosystem
phone-as-key cohort accepts.

### 4.7 / T-007 — Desktop-side root key extraction

**Path.** Attacker gains temporary root on the desktop and reads the
bond key out of the kernel keyring (`keyctl read`) or the libsecret
fallback.

**Mitigation.** **None at the protocol level** for v0.1. Per SPEC §6
and SPEC §3.4, root on the host is post-compromise; the bond key is
no more valuable to an attacker who is already root than the user's
existing session is.

**Code citation.**
- `crates/syauth-core/src/secrets.rs:54-100` (kernel keyring +
  libsecret backends; both readable by root by design).
- `crates/syauth-core/src/bond.rs:58-66` (bond *file* mode 0o600 and
  parent-dir mode 0o700; defense in depth against non-root local
  users, not against root).

**Test.**
- `crates/syauth-core/src/bond.rs::tests::saved_file_mode_is_0o600`.
- `crates/syauth-core/src/bond.rs::tests::parent_directory_mode_is_0o700_after_save`.

**Status.** **Accepted residual** for root-on-host (T-007-RES).
**Mitigated** for non-root local users by file permissions; that is
the in-scope half of this threat.

### 4.8 / T-008 — Denial of unlock (BLE jam)

**Path.** Attacker actively jams the 2.4 GHz band, preventing the
desktop from ever reaching the phone. The user is locked out.

**Mitigation.** The PAM module returns `PAM_AUTHINFO_UNAVAIL` (not
`PAM_AUTH_ERR`) on any transport-side unreachability, which lets the
PAM stack fall through to the next module — typically `pam_unix`
(password). SPEC D7 explicitly carries the password-fallback story.
The install helper inserts syauth at `required` *before* `pam_unix`,
which preserves the fallback semantics.

**Code citation.**
- `crates/syauth-pam/src/auth.rs:312-319` (`TransportError::Unreachable`
  → `AuthInfoUnavail { reason: "offline" }`).
- `crates/syauth-pam/src/auth.rs:319-323` (`TransportError::Timeout`
  → `AuthInfoUnavail { reason: "response-timeout" }`).
- `crates/syauth-cli/src/install_pam.rs:38` (the default control
  flag, which preserves a downstream `pam_unix` line).

**Test.**
- `crates/syauth-pam/tests/pam_e2e.rs::tc02_offline_scenario_returns_authinfo_unavail_under_budget`.

**Status.** **Mitigated** at the protocol layer; the deployment
contract is the documented presence of `pam_unix` (or another
fallback) after the syauth line. `docs/security.md` calls this out
in the operational-hygiene checklist.

### 4.9 / T-009 — Presence inference / tracking

**Path.** Either device advertises a stable, user-linkable identifier
on BLE; a passer-by with a scanner correlates "this UUID = this
person" and tracks the user across visits.

**Mitigation.** The desktop is the *advertiser* (SPEC D8) and
advertises a *rotating* session UUID derived as
`HKDF(bond_key, "syauth-session-v1")[..16]` over the integer minute
counter. The UUID changes every minute and is unlinkable across
minutes without the bond key. The phone is the *scanner*, so it
emits no stable BLE id at all.

**Code citation.**
- `crates/syauth-transport/src/bluez.rs:57` (`SESSION_UUID_ROTATION_INTERVAL`).
- `crates/syauth-transport/src/bluez.rs:62` (`HKDF_INFO_SESSION_V1`).
- `crates/syauth-transport/src/bluez.rs:143` (`session_uuid_for`).

**Test.**
- `crates/syauth-transport/src/bluez.rs::tests::session_uuid_for_is_deterministic_per_minute`.
- `crates/syauth-transport/src/bluez.rs::tests::session_uuid_for_rotates_each_minute`.

**Status.** **Mitigated.**

### 4.10 / T-010 — Timing side-channel on tag / signature verify

**Path.** Verifier uses a non-constant-time comparison; an attacker
on a noisy LAN measures verification time across forged tags to
binary-search the correct tag byte by byte.

**Mitigation.** MAC verification is routed through
`subtle::ConstantTimeEq::ct_eq` over the fixed-length 16-byte tag
arrays. Signature verification is delegated to
`ed25519_dalek::VerifyingKey::verify_strict`, which uses the
constant-time scalar arithmetic of the underlying `curve25519-dalek`.

**Code citation.**
- `crates/syauth-core/src/mac.rs:42-72` (`compute_tag` and
  `verify_tag`; the latter uses `computed.ct_eq(tag)`).
- `crates/syauth-core/src/sign.rs:82-86` (`verify_frame` →
  `pubkey.verify_strict(&body, sig)`).

**Test.**
- `crates/syauth-core/src/mac.rs::tests::verify_is_constant_time_smoke`
  (the timing claim itself rests on the `subtle` contract; the test
  is a behavioural smoke that the constant-time path returns false
  for a known-bad tag).
- `crates/syauth-core/src/mac.rs::tests::verify_rejects_bit_flipped_tag`.
- `crates/syauth-core/src/sign.rs::tests::verify_rejects_bit_flipped_signature`.

**Status.** **Mitigated** at the comparison level. The full
side-channel surface (cache, branch prediction, EM emanations) is
out of scope per SPEC §6.

### F-001 — PAM panic crosses FFI boundary

A Rust panic that escapes the `pam_sm_*` entry point would unwind into
the C frame of libpam, which is undefined behaviour at the language
boundary. We wrap every entry point in `catch_unwind` and translate a
caught panic to `PAM_AUTH_ERR` (fail closed).

- File: `crates/syauth-pam/src/entry.rs:132-143` (`run_entry`).
- Test: `crates/syauth-pam/src/entry.rs::tests::run_entry_catches_panic_and_returns_auth_err`.

### F-002 — Bond file readable by non-root

If the bond file `/var/lib/syauth/bonds.toml` were world-readable, a
co-located user could enumerate every bonded phone (peer ids,
friendly names) and potentially target one for revocation-then-rebind
mischief. We enforce 0o600 on the file and 0o700 on the parent
directory at every save, and the load path *rejects* a parent
directory with looser permissions (returning a typed
`BondError::DirectoryPermissionsTooLoose`).

- File: `crates/syauth-core/src/bond.rs:57-72` (constants),
  `crates/syauth-core/src/bond.rs:412-435` (enforcement).
- Tests: `crates/syauth-core/src/bond.rs::tests::saved_file_mode_is_0o600`,
  `crates/syauth-core/src/bond.rs::tests::parent_directory_mode_is_0o700_after_save`.

### F-003 — Audit-log gap

Every PAM call appends one line to `last.log` (per-call audit trail)
and emits one syslog line under `LOG_AUTHPRIV` with tag
`pam_syauth`. Without this, a forensic investigator could not tell
whether syauth was even reached on a given login attempt. No log
line contains the bond key, the nonce, or the signature bytes (by
inspection of `entry.rs` and `auth.rs`).

- File: `crates/syauth-pam/src/auth.rs:538-555` (`append_last_log`),
  `crates/syauth-pam/src/entry.rs:60-109` (syslog facility and tag).
- Tests: `crates/syauth-pam/src/auth.rs::tests::last_log_append_writes_one_line`,
  `crates/syauth-pam/src/auth.rs::tests::last_log_records_failure_with_unknown_peer`.

### F-004 — `--scripted-oob` bypass goes silent

The pair-flow CLI grew a hidden `--scripted-oob <hex>` flag in S-019
so the emulator-based e2e harness could drive pairing without a
human. Risk: an attacker (or a careless operator) discovers the flag
and uses it to silently bypass the interactive OOB confirmation.

Mitigation: the flag is `hide = true` in clap (not visible in
`--help`), and every use prints `SCRIPTED_OOB_WARNING` (which
includes the phrase "interactive OOB confirmation bypassed") on the
writer so even a CI log review shows the bypass. The flag does NOT
skip the LESC adapter check or the ambiguous-peer check.

- File: `crates/syauth-cli/src/pair.rs:62-72` (constant + min hex
  length), `crates/syauth-cli/src/pair.rs:118-121` (clap arg),
  `crates/syauth-cli/src/pair.rs:495-502` (warning emission).
- Test:
  `crates/syauth-cli/src/pair.rs::tests::scripted_oob_bypasses_prompt_and_warns_without_reading_stdin`.

### F-005 — syauth.conf integrity

`/etc/syauth.conf` is a root-owned config file. A co-located user
who can edit it can flip behaviour. Mitigation: the file is
root:root 0o644 by package convention (operating-system trust
boundary) and the PAM module reads from it via
`Config::from_env`, which has no parsing rabbit holes — the field
set is small (`bonds_file`, `keystore_id_prefix`, `auth_timeout`,
`mock_peer_enabled`), and any malformed value falls back to the
hard-coded default constant.

- File: `crates/syauth-pam/src/config.rs` (the entire module).
- Test: `crates/syauth-pam/src/config.rs::tests::*`.

### F-006 — Wire-format version downgrade

If the parser silently accepted a 0x00 or 0x02 version byte, an
attacker could send pre-v1 or post-v1 frames and either confuse the
core or downgrade to a hypothetical weaker scheme. We pin v1 as the
only accepted version (SPEC §4.5) and reject any other byte with a
typed `FrameError::BadVersion`. The fuzz target in
`crates/syauth-core/fuzz/fuzz_targets/frame_parse.rs` confirms the
parser cannot reach any other admit-state.

- File: `crates/syauth-core/src/frame.rs:31` (the V1 constant),
  `crates/syauth-core/src/frame.rs:182-186` (the version check in
  `decode`).
- Tests: `crates/syauth-core/src/frame.rs::tests::decode_rejects_wrong_version`,
  `decode_rejects_zero_version`,
  `decode_rejects_any_nonzero_version` (proptest).

### F-007 — Bond-store schema downgrade

A future schema (v2) could theoretically be downgraded by an attacker
who replaces the bonds file with a v1 file lacking new safety fields.
The store load path rejects any schema version greater than
`BOND_SCHEMA_VERSION_LATEST` with a typed
`BondError::UnsupportedSchemaVersion`; older versions are accepted
verbatim for forward-compat as designed (SPEC §4.5).

- File: `crates/syauth-core/src/bond.rs:66` (the latest-schema
  constant), and the load path in `crates/syauth-core/src/bond.rs`.
- Tests: covered by the existing bond-store unit tests; schema
  rejection is one of the listed parse paths.

## 7. Accepted Residual Risks

Every risk listed here is **out of scope** for v0.1.0 by an explicit
SPEC anti-goal or by the constraint that v0.1 ships on commodity
hardware without exotic radios.

### T-001-RES — Coerced biometric / synthetic biometric

**Risk.** An attacker who (a) physically forces the user to approve
the BiometricPrompt, or (b) operates a hardware setup that presents a
synthesised fingerprint / face to the phone's sensor, can produce a
fresh signature.

**Rationale for accepting.** This is a rubber-hose attack on Android's
Keystore authentication primitive. syauth piggybacks on whatever
biometric the phone's Keystore offers; we do not redefine it.
v0.1 explicitly does not claim defense against coerced users. v0.2+
candidates for partial mitigation (Wi-Fi RTT distance bounding,
UWB-as-second-factor) are tracked in `specs/syauth/ROADMAP.md` "Out
of roadmap (v0.2 candidates)" — none are committed for v0.1.

### T-006-RES — Phone unlocked at moment of theft

**Risk.** An attacker steals the phone in the few minutes after the
user unlocks it; during that window the BiometricPrompt approval
succeeds (because Android may not require re-auth for the per-key
window) and the attacker unlocks the desktop.

**Rationale for accepting.** This is the residual every Android
phone-as-key system carries. The mitigation is "lock your phone";
the syauth-side surface area is identical to Bitwarden unlock,
Authy push approval, and Apple Auto Unlock on Apple Watch.
`docs/security.md` documents the recommendation in the operational
hygiene checklist.

### T-007-RES — Root on the desktop host

**Risk.** A root attacker on the desktop can read the bond key out
of the kernel keyring (or the libsecret fallback) and forge any
future unlock.

**Rationale for accepting.** Per SPEC §6 T-007 and SPEC §3.4
anti-goal, root on the host is post-compromise; the bond key is no
more privileged than the user's existing session. The mitigation
the user has is `syauth revoke <peer>` from another machine when
the compromise is detected. The bond file and parent directory
modes (0o600 / 0o700) defeat *non-root* local attackers, which is
the in-scope half of this threat.

### T-010-RES — Side channels outside the comparison

**Risk.** Cache-timing, branch-prediction, EM-emanation, or
power-analysis side channels against the bond key or the signing key.

**Rationale for accepting.** Per SPEC §6 T-010, syauth promises
constant-time *byte comparison* and constant-time scalar arithmetic
(both via vetted crates: `subtle` and `curve25519-dalek`). Defeating
deeper side channels would require hardware countermeasures on both
the desktop and the phone, which v0.1 cannot deliver on commodity
hardware. The threat is named here so a deployer running syauth on
a physically-untrusted host can decide whether the residual matches
their threat model.

### T-pairing-physical-RES — Attacker physically present during pairing

**Risk.** An attacker standing within BLE range during `syauth pair`,
holding their own phone advertising the same friendly name, may
appear in the scan results. The operator picks the wrong device.

**Rationale for accepting.** The 6-digit LESC numeric code and the
4-word app-level OOB code must match on both ends. An attacker with
a rogue phone would have to win two unrelated random comparisons.
The protocol cost of mitigating this further (e.g. QR-scan binding
to a one-time URL) is disproportionate for a one-shot pairing
event. Documented in `docs/security.md` as "do not pair in public."

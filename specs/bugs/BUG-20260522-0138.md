# BUG-20260522-0138: SyauthCompanionService zombie after `START_STICKY` resurrect

## Summary
When Android kills the syauth app process under memory/Doze pressure and `START_STICKY` brings `SyauthCompanionService` back, the JVM-static `gattClientFactory` companion-object var is `null` (the entire JVM was torn down). `onCreate` calls `injectClientsForBonds()` which **early-returns when the factory is null**, leaving the resurrected service alive — foreground notification still visible, `isRunning=true`, watchdog logging `service already running; skip` — but with an empty `clients` map. The phone never opens its `BluetoothGatt(autoConnect=true)` link to the desktop, so every challenge from `syauth-presenced` lands on `notifier_slot=None` and instant-fails `transport-error`. The condition persists until the user manually re-opens the app's `MainActivity` (the only place that installs the production factory).

## Reproduction
- Method: failing Robolectric unit test + manual reproduction on the live device
- Test: `SyauthCompanionServiceTest::on_create_installs_default_gatt_client_factory_when_none_preset`
- Command: `JAVA_HOME=/usr/lib/jvm/java-21-openjdk ./gradlew :app:testDebugUnitTest --tests "com.sy.syauth.android.bg.SyauthCompanionServiceTest"`
- Live-host evidence (S25 Ultra, Android, syauth-presenced active since `2026-05-21T22:26:31Z`):
  - `/var/lib/syauth/last.log` shows 121 consecutive `transport-error` rows after the last OK at `1779379665129ms` (`2026-05-21T20:07:45Z`), and zero OKs since.
  - `journalctl --user -u syauth-presenced.service` floods `WARN notify_challenge: notifier_slot=None — phone never subscribed (or task missed event)` once per challenge.
  - `bluetoothctl info 8C:C5:D0:9C:1F:AB` on the host: `Paired=true Bonded=true Trusted=false Connected=false ServicesResolved=false`.
  - `adb shell dumpsys activity services com.sy.syauth.android`:
    - `createTime=-4h30m24s startingBgTimeout=--`
    - `lastActivity=-25m37s restartTime=-2h45m24s createdFromFg=false`
    - The service had been resurrected via `START_STICKY` ~2.75 h before the dump.
  - `adb shell dumpsys bluetooth_manager` GATT Client Map: `Registered AppSize: 0` — i.e., no GATT client process is registered against the Android BT stack at all.
  - `adb logcat -b all -s "syauth.bg.persistent:*" "syauth.bg.watchdog:*"`: only sporadic `watchdog tick: started=false` from the WorkManager periodic worker; **zero** `syauth.bg.persistent` lines despite `PersistentGattClient`'s 15-second reconnect watchdog being supposed to log on every disconnect tick.

- The decisive test: `adb shell am force-stop com.sy.syauth.android && adb shell am start -n com.sy.syauth.android/.MainActivity`. Within 19 seconds the journal printed:
  ```

  01:28:58.659  syauth-presenced: chal_control: Notify event — phone subscribed
  ```
  — confirming the phone-side path was healthy as soon as `MainActivity` re-ran and installed the factory.

## Expected
A `SyauthCompanionService` instance resurrected via `START_STICKY` (without `MainActivity` running first) is **self-sufficient**: its `onCreate` installs a production `GattClientFactory` so the persisted bond's `PersistentGattClient` opens its `autoConnect=true` GATT link without requiring the UI to be launched.

## Actual
The resurrected service sits alive forever with `clients = []` until the user notices unlock has stopped working and manually opens the app. There is no log signal pointing operators at the empty-clients state — the foreground notification, watchdog ticks, and `isRunning=true` all reassure the operator that "the service is running."

## Root Cause
`SyauthCompanionService.gattClientFactory` is a `@Volatile public var` on the service's companion object (`SyauthCompanionService.kt:383–384`). The only production installer is `MainActivity.installPersistentClientFactory` (`MainActivity.kt:457–493`), which runs once when the user opens the activity. Across a process restart, the JVM static is reset to `null`. `injectClientsForBonds()` (`SyauthCompanionService.kt:316–326`) bails on `gattClientFactory ?: return` and silently no-ops. Nothing else in `onCreate` notices.

The original S-011 design assumed `MainActivity` would always be the entry point (it explicitly issues `startForegroundService` for the activity-installed factory to consume). It didn't account for the OS-driven `START_STICKY` resurrection path that Android invokes when it kills the process for memory.

## Fix
Two helpers in `SyauthCompanionService.onCreate` install **every** load-bearing static seam when null:

1. `ensureDefaultGattClientFactory()` — installs the `gattClientFactory` (transport plane). Mirrors `MainActivity.installPersistentClientFactory` field-by-field — same `applicationContext`, same `BluetoothAdapter` lookup, same `PersistentGattClient` construction, same trailing-MAC strip in the `onChallenge` callback, same `PersistentGattClientRegistry.put` for the IPC-driven response path. Without this the `clients` map stayed empty and no GATT connection ever opened.

2. `ensureDefaultCompanionSeams()` — installs the **approval-path seams** that the first iteration of this fix missed. Initial validation revealed that even with the GATT link up, tapping Approve on the phone closed the activity silently without invoking BiometricPrompt because `KeystoreAliasResolver` returned `null`, the activity received `alias=""`, and Keystore (correctly) refused to load a key under an empty alias. Five other static seams are owned by `MainActivity.installCompanionSeams` and need the same default-on-null treatment:
   - `bondKeyProvider` (service seam)
   - `hostnameResolver` (service seam — UI: notification title)
   - `keystoreAliasResolver` (service seam — **load-bearing for Approve to find the signing key**)
   - `challengeVerifier` (service seam — needed before sign)
   - `ChallengeApprovalActivity.responseSink` (**load-bearing for the signed response to reach the host over GATT**)
   - `ChallengeApprovalActivity.cancelSink` (for the Cancel flow)

   Each default reads from `loadPersistedBond(filesDir)` lazily on every lookup, matching the persisted bond record's `bondKey` / `hostName` / `keystoreAlias`. `MainActivity`'s installer remains in place; each assignment becomes idempotent (`!= null` guard short-circuits when the service ran first).

When `BluetoothManager.adapter` is unavailable (e.g., BT toggled off at the OS level), `ensureDefaultGattClientFactory` logs `WARN ensureDefaultGattClientFactory: no BluetoothAdapter; clients will stay empty until MainActivity installs a factory` and returns cleanly — operators have a grep-able breadcrumb instead of silent failure.

`ChallengeApprovalActivity.historyDispatcher` is **not** defaulted: history notifications are UI-tier and not needed for unlock; they re-attach when the user next opens MainActivity.

## Live-host validation
Sequence on the live S25 Ultra, after the apk install:
1. `adb shell run-as com.sy.syauth.android kill -9 <pid>` — simulates an OOM/Doze kill while no Activity is foregrounded.
2. Android's ActivityManager logs `Scheduling restart of crashed service com.sy.syauth.android/.bg.SyauthCompanionService in 1000ms`.
3. New process resurrects 1 s later; `onCreate` runs in a fresh JVM with all static seams null.
4. Both helpers fire visibly: `ensureDefaultGattClientFactory: installed (MainActivity had not yet)` then `ensureDefaultCompanionSeams: installed missing seams (MainActivity had not yet)`. `onCreate: foreground up, clients=1`.
5. Phone opens `BluetoothGatt(autoConnect=true)` → `STATE_CONNECTED` within ~9 s → CCCD subscribe → host journal eventually emits `Notify event — phone subscribed` after BlueZ's stale-subscription rebuild cycle.
6. `sudo` triggers a PAM challenge:


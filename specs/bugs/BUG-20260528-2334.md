# BUG-20260528-2334: phone GATT client wedges in never-completing autoConnect; desktop unlock denied

## Summary
After the desktop daemon re-registers its GATT app (re-pair, restart, or
range-out), the Android `PersistentGattClient` can be left in a
never-completing `connectGatt(autoConnect=true)` background scan that is
never retried, so the desktop's `notifier_slot` stays `None` and every
`sudo` unlock instant-fails (`transport-error` → "unlock denied").

## Reproduction
- Method: live (desktop daemon + real phone over adb) + Robolectric regression test
- Test: `syauth-android/app/src/test/kotlin/com/sy/syauth/android/bg/PersistentGattClientTest.kt::start_arms_reconnect_watchdog_without_state_callback`
- Evidence (live):
  - Desktop daemon (healthy): advertising connectable (`ActiveInstances=1`),
    GATT app registered for peer `56e5d44…`, UUIDs rotating. Every unlock:
    `WARN notify_challenge: notifier_slot=None — phone never subscribed`
    then `tx … outcome=transport-error t_start_ms==t_end_ms`.
  - Phone stack: `LE Gatt clients controlling AutoActiveMode:
    BluetoothLeScanFilter[ Results=0 DeviceAddress=50:BB:B5:B9:93:AB … ]`
    (×6) — the client is scanning for the desktop MAC but never matches,
    and emits **no** `conn state` / `services discovered` logs.
  - `adb shell am force-stop` + relaunch (fresh `PersistentGattClient.start()`)
    connected and subscribed in ~2 s:
    `conn state new=2` → `gatt.refresh()=true; discovering` →
    `services discovered n=5` → `subscribed: writeDescriptor=true`, and the
    daemon then logged `chal_control: Notify event — phone subscribed`.

## Expected Behavior
A `PersistentGattClient` whose initial `connectGatt(autoConnect=true)`
does not complete (the desktop was mid-re-registration, briefly out of
range, or Android's low-duty-cycle background scan stalled) must retry on
its own within `RECONNECT_INTERVAL_MS`, with no user action.

## Actual Behavior
The reconnect watchdog (`reconnectRunnable`, 15 s) was armed **only**
inside `onConnectionStateChange` on `STATE_DISCONNECTED`. A
`connectGatt(autoConnect=true)` that never completes emits **no**
state-change callback at all, so `STATE_DISCONNECTED` never fires, the
watchdog is never scheduled, and the client wedges forever. Re-pairing
does not help because the foreground service is already running, so
`onCreate` (hence `start()`) does not re-run and the wedged client
persists. The only manual recovery was force-stopping the app.

## Root Cause Analysis
"Timer amnesia": the self-healing recovery timer was gated on an event
(`STATE_DISCONNECTED`) that is not guaranteed to arrive. The very first
connection attempt has no prior "connected then disconnected" transition,
so the gating event is absent on the exact path that needs recovery most
— a stalled initial autoConnect after the desktop re-registered its GATT
application (which BlueZ does not announce via a Service Changed
indication).

## Fix
Arm the reconnect watchdog in `start()` itself (right after the
`connectGatt` handle is stored), independent of any state-change
callback. A fast successful connect cancels the pending watchdog via
`STATE_CONNECTED`; a stalled connect is retried after
`RECONNECT_INTERVAL_MS` (a fresh `connectGatt` resets Android's scan
timer — the same effect as the manual force-stop recovery). The
watchdog's own re-arming now flows through `forceReconnect()` →
`start()`, so the previous self-reschedule line is removed to avoid
double-scheduling.

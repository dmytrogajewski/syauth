**Root cause: Samsung One UI on the Galaxy S25 Ultra (Android 15) requires `BLUETOOTH_PRIVILEGED` for `BluetoothLeScanner.startScan(filters, settings, callback)` from non-system apps, despite our app holding `BLUETOOTH_SCAN` (granted, with `neverForLocation` flag) and being in the foreground.** The exception is thrown server-side during `IBluetoothScan.startScan` after `onScannerRegistered` succeeds; this is why our wiring sees `dispatched ok` (the call returned without throwing in our process) but the OS-level `AppScanStats` shows `LE scans (started/stopped): 0 / 0` — the system aborted the scan before it actually started.

`BLUETOOTH_PRIVILEGED` is `signature|privileged` — only platform-signed apps and `/system/priv-app/` apps can hold it. Third-party apps cannot. Verified by changing scan mode from `SCAN_MODE_LOW_LATENCY` to default `SCAN_MODE_LOW_POWER` — same SecurityException. Verified by removing filters — same SecurityException.

Additionally: `dumpsys bluetooth_manager` shows that system apps (`com.samsung.android.mdx.kit`, `com.google.android.gms`) successfully start hundreds of LE scans, while our app records zero — consistent with the privileged-permission gate being the discriminator.

### Architectural implication

The current re-march design (`Phone runs BluetoothLeScanner filtered on the desktop's rotating UUID`) **cannot work as written on this Galaxy S25 Ultra**. Either:

- **Path A (CompanionDeviceManager-driven scan)**: Replace the direct `BluetoothLeScanner.startScan` with `CompanionDeviceManager.associate(AssociationRequest, …)` carrying a `BluetoothLeDeviceFilter(serviceUuid)`. CDM runs the BLE scan under system privileges and presents the user with a system-rendered device picker; on user pick, CDM returns the device and grants the app permission to talk to it without further BLE-scan-permission. This is the Android-blessed path for "find a known device of mine" — and is closer to SPEC §3.2 D8's intent (the OS handles radio-level scanning; the app confirms the OS's pick).
- **Path B (foreground service with FOREGROUND_SERVICE_CONNECTED_DEVICE)**: Start a foreground service during the pair window; some Samsung phones relax BLE-scan restrictions for foreground services with the connected-device type. Existing manifest already declares this permission but no service is started during pair. Empirically uncertain whether this is enough on Samsung One UI Android 15.
- **Path C (drop pair-mode scan; user manually picks via Bluetooth settings)**: Have the user open Android Bluetooth Settings, find the desktop in the system-rendered scan list (which IS privileged), tap to bond, then the app comes in afterwards to do the app-level OOB exchange. Loses the in-app scan experience.

i.e. desktop advertised the slot-N UUID for minute **29650561** — TWO minutes earlier than the phone's `current-1` slot.

`crates/syauth-cli/src/pair_backend.rs` reveals why:
- Line 308 `let minute = unix_minute_floor();` computes the minute ONCE at advertise-start.
- Line 311 `Self::build_advertisement(minute)` builds a SINGLE advertisement carrying that minute's UUID.
- Line 329 `tokio_timeout(PAIR_ADVERTISE_ACCEPT_WINDOW, ...)` (where `PAIR_ADVERTISE_ACCEPT_WINDOW = Duration::from_secs(60)`) then waits 60s for a phone-pubkey write; on timeout, returns `PairError::NoPeers` and the desktop process exits.
- The advertised UUID is **never rotated**.

Combined with the phone's filter window of {current, current-1}, the overlap window between desktop pair-mode availability and phone pair-tap is **at most 60 seconds**: the user must tap "Pair with computer" on the phone within ≤60s of running `syauth pair` on the desktop. In our run the gap was ~2.5 minutes, so the desktop had already exited with `NoPeers` before the phone's CDM picker even opened, and even if it had still been advertising, the slot UUIDs would have mismatched.

This explains every "picker stays empty" observation across multiple attempts in this session.

Fix candidates:
1. Make the desktop rotate the advertised UUID every minute (proper fix): on each minute boundary, replace the advertisement with one carrying the new slot's UUID, and extend the accept window from 60s to ~5min.
2. Widen the phone's slot window to {current, current-1, current-2, current+1} so it can find a desktop that's been advertising for up to ~3 minutes.
3. Replace the rotating pair-mode UUID with a single fixed `SYAUTH_PAIR_SERVICE_UUID` (since pair-mode bond_key is zero, the per-minute rotation offers no privacy — just brittleness).

Option (3) is simplest and aligns with the "pair mode is short-lived, no privacy from rotation" framing. Options (1)+(2) are layered defence-in-depth.

Picker-empty root cause is now fully diagnosed. No further probe is needed; the next action is a code change, chosen by the user.

### Probe 4 result — agent callback deadlock (TOKIO-STARVATION FIX 2026-05-17T~20-48)

### Waybar applet — user-visible LESC numeric comparison (2026-05-17T~21-15)

Per SPEC §3.2 D5, the LESC numeric-comparison code must be user-visible on both sides. The desktop already gates the 6-digit code via the BlueZ agent's `request_confirmation` callback, but the existing handlers were either headless (`make_auto_accept_confirm_handler`, `--yes`) or required an interactive terminal (`make_stdio_confirm_handler`). For a normal desktop session (niri + waybar), neither is right: there's no terminal for stdin, and auto-accept defeats the comparison.

Added a third confirmation handler + a waybar applet that surfaces the code on the bar:

**`crates/syauth-cli/src/pair_backend.rs::make_waybar_confirm_handler`** (new):
- Writes `$XDG_RUNTIME_DIR/syauth/pair-request.json` with `{schema_version, kind, request_id, passkey, created_at_secs}`.
- Polls `$XDG_RUNTIME_DIR/syauth/pair-response.json` every 100ms for up to 25s (5s safety margin before BlueZ's 30s SMP_CONN_TOUT).
- Reads `{request_id, decision}`. Matching `request_id` ensures stale responses don't accept a fresh request.
- Cleans up both files and returns `true`/`false`.
- `--waybar` CLI flag selects this handler (mutually exclusive with `--yes`).

**`~/sources/sy/src/syauth.rs`** (new module in the sy bar/rice repo):
- `sy syauth --waybar` reads the request file and emits `{"text":"syauth:<code>","class":"pending","tooltip":"…"}` to waybar. When no request is pending, emits `{"text":"","tooltip":""}` so the bar slot is empty.
- `sy syauth accept` writes `pair-response.json` with `{decision:"accept"}` + the matching `request_id`. Atomic write-then-rename so a partial write never surfaces to the desktop polling loop.
- `sy syauth reject` writes the same with `{decision:"reject"}`.
- `sy syauth status` prints the pending request for CLI debugging.
- Schema-version check (must equal 1); kind check (must equal `pair_confirm`). Unknown values produce an empty slot — defensive against an out-of-date applet "accepting" a future message it doesn't understand.

**`~/sources/sy/configs/waybar/config.jsonc`**:
- New module `custom/sy-syauth` at the leftmost position of `modules-right`.
- 1-second poll interval (fast enough that the code lands on the bar within a frame of the agent callback firing).
- `on-click` → `sy syauth accept`; `on-click-right` → `sy syauth reject`.
